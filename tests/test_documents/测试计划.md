# 测试计划（GB8567—88）

## 1 引言

### 1.1 编写目的
本测试计划用于指导ROS2智能药品拣选系统的测试工作。主要目的是：
- 对系统各个功能模块进行全面测试
- 验证系统是否满足功能需求和质量指标
- 确保AI医疗助手、药房后端、权限认证等各组件的集成质量
- 识别和记录系统缺陷，为质量改进指导方向

本计划适用于系统开发团队、质量保证部门及项目管理人员。

### 1.2 背景
**系统名称**：ROS2智能药品拣选系统（agent_with_backend）

**项目历史与现状**：
- 本项目是一个集成智能医疗助手与自动化药房后端的系统
- 系统包含多个模块：AI医疗助手（P1/Agent模块）、自动化药房后端API、权限认证系统、智能筛选系统、ROS2集成
- 核心功能包括药品库存管理、智能用药推荐、医生审批工作流、机器人自动化取药
- 已完成的工作：基础架构搭建、核心API开发、P1医疗助手系统、权限认证系统、部分集成测试

**测试环境要求**：
- Python 3.8+ 环境
- 一个配置的LLM API密钥（OpenAI或Claude）
- SQLite数据库
- ROS2环境（可选，用于ROS集成测试）
- Flask服务器运行环境

### 1.3 定义
- **LLM**：大语言模型（Large Language Model）
- **API**：应用程序接口（Application Programming Interface）
- **RBAC**：基于角色的访问控制（Role-Based Access Control）
- **ROS2**：机器人操作系统第二版（Robot Operating System 2）
- **REST**：表现层状态转化（Representational State Transfer）
- **端到端测试**：E2E测试，从用户操作到系统响应的完整流程测试
- **集成测试**：验证不同模块间的协作正确性
- **单元测试**：对单个函数或类的独立测试
- **CRUD**：创建（Create）、读取（Read）、更新（Update）、删除（Delete）操作

## 3 测试设计说明

### 3.1 测试1：单元测试 - 药品API基础操作（UT-001）

说明对第一项测试内容的测试设计考虑。

#### 3.1.1 控制

**控制方式**：完全自动化测试
- **输入方式**：自动生成测试数据，通过Python代码直接调用被测函数
- **控制顺序**：按测试类别组织，单一职责原则，一个函数测一个功能
- **结果记录方式**：pytest自动记录、生成HTML报告，结合覆盖率工具

**具体步骤**：
1. 使用pytest discovery自动发现测试文件
2. 对每个测试函数执行前置固定装置（setup）初始化测试环境
3. 执行测试逻辑，记录任何异常
4. 对每个测试函数执行后置固定装置（teardown）清理环境
5. 生成覆盖率报告

#### 3.1.2 输入

**输入数据说明**：
- **药品数据**：
  - 标准药品：名称、分类、库存量、有效期、货架位置
  - 边界数据：最小库存、最大库存、过期日期为今天的药品
  - 异常数据：重复ID、特殊字符名称、无效库存（负数）

- **操作指令**：
  - 有效操作：创建(CREATE)、读取(READ)、更新(UPDATE)、删除(DELETE)
  - 边界操作：批量操作、并发操作、操作不存在的药品
  - 异常操作：未授权操作、超出限制的操作

**数据选择策略**：
- 使用等价类划分：有效输入类、边界输入类、无效输入类
- 采用边界值分析：对于数值类字段取边界值
- 随机生成一部分测试数据以发现未预见的问题

#### 3.1.3 输出

**预期输出**：
- **成功操作**：
  - 返回HTTP 200-201状态码

## 6 风险分析与应对

### 6.1 主要风险识别

| 风险ID | 风险描述 | 影响度 | 概率 | 应对方案 |
|--------|--------|------|------|--------|
| R1 | 外部LLM API服务不可用 | 高 | 中 | 使用mock LLM，测试本地逻辑；提前准备备用LLM供应商 |
| R2 | ROS2环境配置复杂 | 中 | 高 | 提前搭建ROS2测试环境；准备详细配置文档 |
| R3 | 数据库锁定导致测试并发问题 | 中 | 中 | 使用内存数据库或隔离测试实例；增加测试超时 |
| R4 | 团队成员缺乏系统设计的深入理解 | 中 | 中 | 组织架构讲座；分配有经验的开发人员指导 |
| R5 | 测试环境与生产环境差异 | 低 | 中 | 使用Docker容器保证环境一致性；定期同步配置 |

### 6.2 应急方案

- **若LLM API不可用**：使用Mocking框架（unittest.mock）模拟LLM返回
- **若ROS2环境故障**：在日志模式（LOG_ONLY）下进行功能验证
- **若时间限制**：优先完成P1、P2模块的测试，P3模块可延期
