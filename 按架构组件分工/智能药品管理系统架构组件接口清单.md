# 智能药品管理系统 - 架构组件接口清单

## 📋 概述

基于《智能药品管理系统 - 五人团队分工方案（按架构组件分工）》文档，本文档定义了各架构组件间需要确定的技术接口清单。按照方案要求，所有组件接口契约需要在第4周前正式确定。

## 🔗 组件依赖关系图

```
数据库与基础架构（组件1）
    ↓（提供基础服务）
┌─────────────────────────────────────────┐
│       所有其他组件（2,3,4,5）           │
└────────────────────────┬────────────────┘
    │                    │                    │
    ↓（提供药品数据）    ↓（提供筛选服务）    ↓（提供认证服务）
药品管理API（组件2）← 智能筛选系统（组件3）← 权限认证系统（组件4）
    │                    │                    │
    └────────────────────┴────────────────────┘
                         ↓（提供数据展示）
                 前端界面开发（组件5）← 调用所有组件API
```

## 📊 接口确定时间表

|------|----------|----------------|

| **阶段2** | 第2周结束前 | 1. 权限模型和RBAC接口定义<br>2. 智能筛选服务接口定义<br>3. 前后端数据协议和组件接口 |
| **阶段3** | 第4周结束前 | 1. 所有核心API接口契约正式确定<br>2. 集成测试方案和端到端流程定义 |

### 组件2：药品管理API开发

#### 需要提供的API接口

| 接口类别 | 接口端点 | HTTP方法 | 描述 | 请求参数 | 响应格式 |
|----------|----------|----------|------|----------|----------|
| **药品查询** | `/api/drugs` | GET | 获取药品列表（支持分页、筛选、排序） | `page`, `limit`, `category`, `name`, `sort_by` | `{success, drugs[], pagination}` |
| | `/api/drugs/{id}` | GET | 获取单个药品详情 | `id` | `{success, drug}` |
| **药品管理** | `/api/drugs` | POST | 创建新药品 | 药品完整信息（30+字段） | `{success, drug_id}` |
| | `/api/drugs/{id}` | PUT | 更新药品信息 | `id`, 更新字段 | `{success, message}` |
| | `/api/drugs/{id}` | DELETE | 删除药品 | `id` | `{success, message}` |
| **库存管理** | `/api/inventory` | GET | 获取库存信息 | `drug_id`, `low_stock`, `expiry_soon` | `{success, inventory[]}` |
| | `/api/inventory/{drug_id}/adjust` | POST | 调整库存数量 | `drug_id`, `quantity`, `type`, `reason` | `{success, new_quantity}` |
| **分类管理** | `/api/categories` | GET | 获取药品分类列表 | `parent_id`, `level` | `{success, categories[]}` |
| | `/api/categories` | POST | 创建新分类 | `name`, `description`, `parent_id` | `{success, category_id}` |
| **搜索功能** | `/api/search/drugs` | GET | 综合搜索药品 | `keyword`, `filters`, `facets` | `{success, drugs[], facets}` |
| **批量操作** | `/api/batch/drugs` | POST | 批量药品操作 | `operation`, `drug_ids[]`, `parameters` | `{success, job_id}` |

#### 需要调用的外部接口
- **组件1（数据库架构）**：需要数据库Schema和基础规范接口
- **组件4（权限认证）**：需要调用认证API验证用户权限
- **组件3（智能筛选）**：提供药品数据给筛选系统

## 🎯 需要优先确定的核心接口

### 第一阶段（第1周确定）

#### 1. 数据库Schema接口
| 模型 | 字段定义 | 关联关系 |
|------|----------|----------|
| **药品模型** | `id`, `name`, `category`, `specification`, `price`, `stock`, `expiry_date`等30+字段 | 与分类、库存关联 |
| **用户模型** | `id`, `username`, `email`, `role`, `status`, `created_at` | 与角色、权限关联 |
| **权限模型** | `role_id`, `permission_code`, `resource`, `action` | RBAC权限控制 |
| **症状模型** | `symptom_id`, `standard_name`, `synonyms[]`, `category` | 与药品关联 |

#### 2. API基础规范接口
| 规范类型 | 具体约定 |
|----------|----------|
| **认证机制** | JWT令牌格式、刷新机制、有效期（24小时访问token，7天刷新token） |
| **错误处理** | 统一错误码（如AUTH_001、DRUG_002）、HTTP状态码映射（401未授权，403禁止访问） |
| **分页规范** | `page`（默认1）、`limit`（默认20，最大100）、分页响应格式 |
| **时间格式** | ISO 8601标准时间格式（YYYY-MM-DDTHH:mm:ssZ） |
| **响应格式** | `{success: boolean, data: any, message?: string, error_code?: string}` |

### 第二阶段（第2周确定）

#### 1. 组件间关键接口
| 接口 | 提供组件 | 调用组件 | 接口定义 |
|------|----------|----------|----------|
| **认证中间件接口** | 组件4 | 所有组件 | JWT验证、权限检查中间件接口 |
| **药品查询接口** | 组件2 | 组件3,5 | `GET /api/drugs` 详细参数和筛选条件定义 |
| **筛选服务接口** | 组件3 | 组件5 | `POST /api/screening/query` 请求响应格式定义 |
| **前端数据协议** | 组件5 | 所有组件 | API调用封装、错误处理、数据转换接口 |

#### 2. 权限模型接口
| 模型组件 | 接口定义 | 权限控制 |
|----------|----------|----------|
| **RBAC模型** | 角色-权限-用户三级权限控制 | 管理员、医生、药剂师、普通用户角色定义 |
| **资源权限** | 药品CRUD、筛选查询、用户管理等资源权限 | 细粒度权限控制（如create:drug, read:drug, update:drug） |
| **API权限** | 各API端点权限验证规则 | 基于角色的API访问控制矩阵 |

### 第三阶段（第4周前最终确定）

#### 所有组件完整接口契约
1. **组件1完整技术文档**：数据库Schema、API规范、基础架构接口
2. **组件2完整API文档**：所有药品管理相关接口（CRUD、库存、分类、搜索）
3. **组件3完整API文档**：所有智能筛选相关接口（症状处理、筛选查询、配置管理）
4. **组件4完整API文档**：所有用户权限相关接口（认证、用户管理、权限控制、审计）
5. **组件5完整技术文档**：前端数据协议、组件接口、路由定义、UI规范

## 🔧 接口测试和验证要求

### 接口测试清单
| 测试类型 | 测试内容 | 验证标准 |
|----------|----------|----------|
| **连通性测试** | 组件间接口是否能正常通信 | HTTP状态码200，返回标准格式，响应时间<500ms |
| **参数验证** | 必填参数、参数类型、参数范围验证 | 正确参数通过，错误参数返回明确错误信息 |
| **权限验证** | 接口权限控制是否生效 | 无权限访问返回403，有权限正常访问，角色权限正确映射 |
| **性能测试** | 接口响应时间、并发处理能力 | 单接口响应时间<1s，支持100+并发请求 |
| **错误处理** | 异常情况下的错误返回 | 返回标准错误格式，包含错误码和用户友好消息 |

### 集成测试场景
1. **用户登录 → 药品查询**：验证认证和药品API集成
2. **症状筛选 → 药品详情**：验证筛选和药品详情API集成
3. **权限变更 → API访问**：验证权限系统与各组件集成
4. **前端页面 → 后端API**：验证前端数据协议与后端API集成
5. **完整业务流程**：用户登录 → 症状筛选 → 查看药品 → 提交审批 → 数据导出

## 📄 接口文档管理

### 文档格式要求
1. **OpenAPI/Swagger规范**：所有REST API必须提供OpenAPI 3.0规范文档
2. **TypeScript接口定义**：前端组件必须提供完整的TypeScript接口定义
3. **接口变更日志**：记录接口变更历史，维护版本兼容性
4. **示例代码**：提供主要语言的调用示例（Python、JavaScript、TypeScript）
5. **错误码列表**：完整的错误码定义和解释文档

### 版本管理策略
1. **API版本前缀**：所有API使用版本前缀，如`/api/v1/drugs`
2. **向后兼容**：新增字段不破坏现有客户端，废弃字段标记`deprecated`
3. **版本迁移计划**：重大变更提供迁移路径和时间表（至少2周迁移期）
4. **组件版本同步**：各组件版本号同步更新，确保兼容性

## 📌 总结

本接口清单共定义了**5个架构组件**的**核心技术接口**，涵盖了智能药品管理系统的全部技术需求。按照时间表分阶段确定，确保组件并行开发顺利进行。

**关键成功因素**：
1. **接口契约优先**：先定义接口，再实现内部逻辑
2. **组件独立开发**：各组件在接口契约下独立开发，减少耦合
3. **持续集成验证**：每日验证组件接口兼容性
4. **文档驱动开发**：所有接口必须有完整技术文档
5. **版本管理严格**：确保接口变更的向后兼容性
