# 文档更新与整合设计方案

## 1. 概述

### 1.1 项目背景
当前项目文档存在以下问题：
1. **团队项目说明文档（v3.0）**：部分内容未反映实际项目状态，基于早期规划而非当前实现
2. **文档分散**：API参考、集成指南和故障排除分为三个独立文档，查找不便
3. **信息重复**：多个文档包含相同的环境配置和安装说明
4. **读者定位不明确**：不同角色（开发者、管理者、使用者）需要阅读多个文档

### 1.2 设计目标
1. **准确性**：完全基于实际项目状态重写团队项目说明文档，反映Flask后端和实际完成情况
2. **完整性标注**：明确标注各角色实际完成状态，区分已完成、部分完成和未完成部分
3. **实际API参考**：基于实际Flask API端点（`/api/*`）更新技术参考，而非规划中的FastAPI端点
4. **整合性**：将三个技术文档（API参考、集成指南、故障排除）合并为单一综合文档
5. **实用性**：按用户场景和问题类型组织内容，提高查找效率
6. **可维护性**：创建清晰的结构，便于未来更新和维护

### 1.3 核心约束
- ✅ **必须保留**：所有技术细节、API端点定义、故障排除步骤
- ✅ **必须更新**：团队项目说明文档以反映实际项目状态
- ✅ **必须整合**：三个技术文档为一个逻辑连贯的综合文档
- ✅ **必须面向**：开发者、项目管理者和系统使用者三类读者
- ⚠️ **不能删除**：任何有价值的配置、API或故障排除信息
- ⚠️ **保持原意**：所有技术内容保持原意，仅进行重组和优化

### 2.2 当前项目实际状态
基于对项目代码和README.md的分析，实际项目状态如下：

#### 2.2.1 项目名称
**ROS2-based Intelligent Medicine Pickup System**（ROS2智能药品拣选系统）

#### 2.2.2 实际技术栈
- **后端服务**：Flask + SQLite + ROS2 (rclpy)
- **AI助手**：Python + Claude/OpenAI/DeepSeek API + 异步工具调用
- **通信协议**：HTTP REST API + ROS2话题
- **数据库**：SQLite（pharmacy.db）

#### 2.2.3 核心功能（已实现）
1. **智能药房管理**：药品库存、有效期管理、货架定位
2. **ROS2集成**：自动化取药任务发布到仿真系统
3. **AI医疗助手**：症状分析、药品推荐、剂量计算
4. **医生审批流程**：完整的电子审批工作流
5. **定时任务**：自动过期药品清扫和库存管理

#### 2.2.4 项目实际结构
与文档v3.0描述有差异，实际结构在README.md中有详细描述，包括：
- `backend/`：智能药房后端服务（Flask）
- `P1/`：AI医疗助手系统
- `scripts/`：实用脚本
- `docs/`：文档目录
- `tests/`：测试文件

#### 2.2.5 各角色实际完成状态
基于对项目代码的详细分析，各角色完成状态如下：

| 角色 | 文档v3.0描述 | 实际完成状态 | 主要产出文件状态 | 关键差异 |
|------|-------------|-------------|-----------------|----------|
| **P1**（Agent核心） | 1500行代码，Agent循环、工具注册、LLM调用 | ✅ **已完成** | `core/agent.py`、`tools/registry.py`、`llm/client.py`等完整 | 实际架构更完善，代码量约6,600行 |
| **P2**（医疗工具） | 2900行代码，医疗工具实现、数据库集成 | ⚠️ **部分完成** | `drug_db.py`、`tools/inventory.py`集成后端，`tools/medical.py`仍为mock实现 | `fill_prescription`工具仍为mock |
| **P3**（规划子代理） | 1300行代码，规划器、症状提取子代理 | ❌ **未完成** | 无`planner.py`、无`subagents/`目录 | 规划器和子代理未实现 |
| **P4**（后端API） | 2100行代码，FastAPI后端 | ⚠️ **完成但不同** | `backend/`目录完整，但使用Flask（非FastAPI） | 技术栈不同（Flask vs FastAPI），API端点不同 |
| **P5**（前端） | 3300行代码，5个HTML页面 | ❌ **未完成** | 无`web/`目录，无HTML/CSS/JS文件 | 前端界面未开发 |
| **P6**（测试审批） | 1450行代码，审批模块、审计日志 | ⚠️ **部分完成** | `backend/approval.py`存在且完整，但无`audit_logger.py`、`auth.py` | 审批模块在backend中实现，审计日志未完成 |

#### 2.2.6 实际API端点与规划差异
- **实际Flask端点**（在`backend/app.py`中实现）：
  - `/api/health` - 健康检查
  - `/api/drugs` - 药品管理
  - `/api/order` - 批量取药
  - `/api/approvals/*` - 审批管理
  - `/api/orders` - 订单历史

- **文档v3.0规划端点**（未实现）：
  - `/chat` - 聊天接口
  - `/approve` - 审批接口（实际为`/api/approvals/{id}/approve`）
  - `/pending` - 待审批列表（实际为`/api/approvals/pending`）
  - `/prescription/{id}` - 处方查询
  - `/admin/*` - 管理员接口

- **集成状态**：P1的`drug_db.py`和`tools/inventory.py`已通过`PharmacyHTTPClient`连接实际后端API
